# EdgeAlignInspect 项目优化检查说明

检查时间：2026-05-08

## 1. 当前结论

项目当前主体目录仍然按职责分层：`Forms`、`Controls`、`Geometry`、`Models`、`Processing`、`Sdk`、`Lib`。项目文件是 VS2019 可加载的旧式 `.csproj`，目标框架是 `.NET Framework 4.7.2`，`PlatformTarget` 为 `x64`，`Prefer32Bit=false`，没有再改成 x86/x32。

我做了独立输出目录构建验证，`dotnet build` 可以通过。对默认 `bin\Debug` 做 `Rebuild` 时失败的原因不是代码编译错误，而是 `EdgeAlignInspect.exe` 和 `halcondotnet.dll` 被当前运行中的 `EdgeAlignInspect` 进程和 VS 远程调试器锁定。关闭正在运行的程序后再重建即可。

但是，项目仍有一个必须优先处理的健康问题：大量 `.cs` 文件前部存在约 4KB 的二进制前缀，`rg` 和 Roslyn 编译器仍能识别后面的源码，所以能编译；但 PowerShell、部分编辑器、VS 设计器或代码格式化工具会把它们显示成乱码。这会影响维护、搜索、合并、代码审查和长期交付，建议作为 P0 处理。

## 2. 页面层检查

相关文件：`Forms\Form1.cs`、`Forms\Form2.cs`

现状：

- 主配置页集中在 `Form1.cs`，包含 ROI 编辑、参数同步、运行检测、结果展示、控件初始化，文件体量很大。
- 页面目前已经有“圆外扩”参数，绑定到 `numBaseOutward`，只在选中圆基准 ROI 时启用。
- 基准卡尺和检测卡尺都开放了点数、Sigma、阈值、极性等参数。
- 模板匹配默认在构造函数里关闭，避免加载图片后自动出现模板 ROI，这一点符合当前使用习惯。
- 页面使用固定右侧宽度和大量 `SetBounds` 绝对布局，短期稳定，但对 DPI、字体、不同屏幕比例不够友好。

主要问题：

- `Form1.cs` 同时承担 UI、状态同步、Job 构建、算法调用、结果绘制，后续新增功能会越来越难定位问题。
- `Forms\Form1.Designer.cs`、`Forms\Form2.Designer.cs` 仍残留在目录里，但当前项目文件没有引用它们，而且它们也存在二进制前缀。VS 如果开启“显示所有文件”，会让人误以为设计器文件还有效。
- “阈值”虽然已经开放，但对圆基准来说语义不够明确。用户看到的是基准/检测阈值，实际圆基准拟合也复用了基准侧参数，容易误判为没有开放圆阈值。
- 当前结果区能显示 OK/NG 和统计，但缺少“圆拟合质量”的直观反馈，例如有效边点数量、拟合残差、半径偏差、被剔除点数量。

建议：

- 将 `Form1` 拆成三层：页面控件初始化、UI 与 Job 参数同步、运行与结果展示。至少先把 `BuildJobForReturn`、`ApplyAutoCaliperGeometry`、`ShowResult`、卡尺参数同步拆到独立 helper。
- 把“圆外扩”旁边的“阈值”在选中圆基准时显示为更明确的“圆阈值/边缘阈值”，或者增加一组独立的“圆基准卡尺参数”标题。
- 在画布上显示圆拟合采样点，区分成功点、失败点、最终拟合圆。这样可以直接判断“不贴光斑”是阈值、极性、ROI 半径、外扩距离还是采样点分布造成的。
- 删除或迁移未被项目引用的旧 Designer/拆分前文件，避免 VS 资源管理器中出现无效文件。

## 3. 算法层检查

相关文件：`Processing\TemplateEdgeInspectProcessor.cs`

现状：

- 直线基准使用 HALCON metrology line measure。
- 圆基准已改为径向卡尺搜索：从圆外侧扩展距离开始，沿圆法向向圆心内部搜索。
- 圆基准默认 `SearchOutward=6.0`，线基准和检测默认 `SearchOutward=0.0`。
- 圆基准调用 `MeasurePos` 后会选择更靠外的圆边点参与拟合，方向上符合“从圆外往圆心内部找边”的要求。

主要问题：

- 圆拟合是否“贴光斑”，不只由阈值决定，还和 ROI 初始半径、外扩距离、卡尺长度、卡尺宽度、Sigma、极性、采样点数量共同相关。
- 当前圆基准默认 `Transition=all`，在光斑边缘附近有高光、阴影、毛边或相邻结构时，容易拾取到非目标边。对于“外部暗、内部亮”的光斑，建议允许明确设置圆搜索极性，并提供推荐默认值。
- `Threshold` 目前只是强度边缘筛选阈值，没有配合质量指标。阈值太高可能找不到弱边；阈值太低可能吃到噪声或外圈虚边。
- 缺少圆拟合质量门槛，例如最少有效点比例、最大残差、半径变化范围。这会导致“能拟合出圆，但不是贴着目标光斑”的情况仍被当作有效结果。
- 异常处理多数会吞掉底层原因，最终只给失败提示。调试阶段需要更多上下文，例如 ROI 名称、采样点数、阈值、极性、有效边点数。

建议：

- 为圆基准增加独立质量参数：`MinCircleEdgePoints`、`MaxCircleFitResidualPx`、`MaxRadiusDeltaPx`。
- 结果里增加 `CircleEdgePointCount`、`CircleFitResidualPx`、`FittedRadiusPx`，并在 UI 显示。
- 圆基准极性从“任意”改为可配置且有明确推荐：外暗内亮时使用外到内方向对应的单一极性，降低误检。
- 对圆拟合增加调试 overlay：采样卡尺线、命中点、未命中方向、最终圆。这个对现场调参最有价值。
- 阈值需要继续开放，但应和 Sigma、极性、外扩距离一起作为“圆基准卡尺参数”开放，而不是只暴露一个阈值。

## 4. 模型与参数层检查

相关文件：`Models\EdgeInspectJob.cs`、`Models\CaliperParameters.cs`

现状：

- `EdgeInspectJob` 已有线基准、圆基准、检测 ROI、模板匹配、像素分辨率、外部毛刺公差等字段。
- 每个 ROI 可以拥有独立卡尺参数，这是正确方向。
- `Normalize()` 会补默认值，并过滤空 ROI。
- `CaliperParameters` 包含 `NumMeasures`、`MeasureLength`、`MeasureWidth`、`Sigma`、`Threshold`、`SearchOutward`、`Transition` 等关键字段。

主要问题：

- `Normalize()` 会移除空 ROI，这对运行前清理是合理的；但如果上位机希望保留“未完成配置”的 ROI 草稿，需要额外区分“编辑态 Job”和“运行态 Job”。
- Job 当前是直接可变对象，上位机传入后 SDK 会 deep clone，但如果未来多线程调用，需要进一步明确线程安全边界。
- 参数缺少版本号。后续继续增加圆拟合质量参数、调试参数、单位字段时，旧 Job 反序列化容易出现兼容问题。
- 像素分辨率和外部毛刺公差由 SDK 入口写入，UI 侧没有明显显示。现场调试时会不知道当前判定单位是像素还是物理单位。

建议：

- 给 `EdgeInspectJob` 增加 `SchemaVersion`，后续做兼容升级。
- 区分 `NormalizeForEdit()` 和 `NormalizeForRun()`，避免编辑态数据被运行态清理逻辑意外删除。
- 把圆基准质量参数纳入 `CaliperParameters` 或新建 `CircleFitParameters`，避免把所有算法参数都塞进一个通用卡尺结构。
- UI 结果区显示当前 `PixelResolutionX/Y`、外部公差来源和最终换算后的判定阈值。

## 5. SDK 与上位机对接层检查

相关文件：`Sdk\TemplateEdgeInspectSdk.cs`

现状：

- 对外入口只有两个：`OpenSetupDialog(Bitmap image, EdgeInspectJob currentJob)` 和 `RunInspection(Bitmap image, EdgeInspectJob job, double acceptedTolerance, double pixelResolutionX, double pixelResolutionY)`。
- SDK 会复制 Bitmap 和 Job，避免直接修改调用方对象。
- 参数校验覆盖了空图、空 Job、公差为负、像素分辨率小于等于 0。
- 异常会转换成 `EdgeInspectResult.Success=false`，不会直接抛给上位机。

主要问题：

- `OpenSetupDialog` 是 WinForms 对话框，必须在 STA/UI 线程调用；当前接口说明里没有写清楚。
- `RunInspection` 的异常全部变成字符串，缺少错误码，自动化系统不好做分支处理。
- 上位机如果只想改阈值、极性、圆外扩，需要直接理解 `EdgeInspectJob` 结构；目前没有便捷 API 或配置文档。
- 缺少 Job 的保存/加载示例，尤其是版本升级、单位、ROI 坐标来源、模板匹配数据如何持久化。

建议：

- 增加接口说明文档：调用线程、输入图像生命周期、返回 null 的含义、单位约定、x64 HALCON 运行要求。
- 给 `EdgeInspectResult` 增加 `ErrorCode` 或 `FailureReason` 枚举，区分图像无效、模板匹配失败、基准失败、检测 ROI 失败、参数非法、HALCON 异常。
- 提供 `ValidateJob(job)`，上位机运行前可以先拿到完整配置问题列表。
- 提供 Job JSON 保存/加载示例，并锁定字段版本。

## 6. 构建与部署检查

相关文件：`EdgeAlignInspect.csproj`、`Lib\halcondotnet.dll`

现状：

- `.csproj` 是旧式 MSBuild 项目，可被 VS2019 加载。
- `TargetFrameworkVersion=v4.7.2`。
- `PlatformTarget=x64`，`Prefer32Bit=false`，这对 HALCON 64 位 DLL 是必要的，不能改成 x86/x32。
- HALCON 引用来自 `Lib\halcondotnet.dll`，并设置了 `Private=true`。

主要问题：

- `packages` 目录里有 `HalconDotNet.19.11.0\lib\netcoreapp3.0\halcondotnet.dll`，而项目是 `.NET Framework 4.7.2`。当前实际引用是 `Lib\halcondotnet.dll`，这可以工作，但包目录和 README 容易误导。
- `bin\Debug\net472` 是之前 SDK-style 项目构建留下的输出目录，不属于当前旧式项目的正常输出。
- 运行中的程序会锁住 `bin\Debug\EdgeAlignInspect.exe` 和 `halcondotnet.dll`，导致 Rebuild 失败。
- 当前 README 是乱码，不能作为交付说明。

建议：

- 保持 x64，不再修改为 x86/x32。
- 清理旧的 `bin\Debug\net472` 和无效说明，统一输出目录说明为 `bin\Debug\`。
- 补充部署检查项：目标机必须安装匹配版本的 64 位 HALCON runtime，或者确保所有必要 HALCON 运行 DLL 与许可配置齐全。
- 将 README 改成正常中文，并指向本说明和 SDK 调用示例。

## 7. 优先级建议

P0：

- 修复所有带二进制前缀的 `.cs` 文件，保证源码是纯文本 UTF-8。
- 清理未被 `.csproj` 引用的旧 Designer/拆分前文件。
- 关闭正在运行的 `EdgeAlignInspect.exe` 后做一次干净 Rebuild，确认无缓存依赖。

P1：

- 给圆基准增加拟合质量指标和 overlay。
- 将圆基准阈值、极性、外扩、卡尺长度/宽度作为一组明确参数展示。
- 增加 `ValidateJob` 和错误码。

P2：

- 拆分 `Form1.cs`，降低页面和业务逻辑耦合。
- 增加 Job 版本号和保存/加载示例。
- 完善 README 与上位机对接文档。

## 8. 针对当前“圆不贴光斑”的判断

这个问题很可能和阈值有关，但不是只和阈值有关。更准确地说，圆是否贴边取决于：

- ROI 圆初始半径是否接近光斑真实边缘。
- `SearchOutward` 是否足够覆盖圆外暗区到圆边。
- `MeasureLength` 是否足够从外侧扫到内侧。
- `Threshold` 是否既能压掉噪声，又不丢掉弱边。
- `Sigma` 是否把边缘抹得过宽。
- `Transition` 是否与外到内的明暗变化一致。
- 拟合时是否过滤了错误边点。

所以建议不是只把阈值开放，而是把圆基准的“阈值 + 极性 + 外扩 + 拟合质量”一起开放，并在结果里画出命中的边点。这样现场一眼就能看出圆为什么没有贴住光斑。
